Program 1 - Parallel Fractal Generation Using Threads

Approach. Each worker calls mandelbrotSerial() on a subset of image rows. I first implemented the naïve block decomposition (thread t gets a contiguous band of height/numThreads rows), then replaced it with an interleaved (cyclic) decomposition that fixes load imbalance:

// Final: interleaved rows - thread t computes rows t, t+numThreads, t+2*numThreads, ...
for (unsigned int row = args->threadId; row < args->height; row += args->numThreads)
    mandelbrotSerial(x0,y0,x1,y1, width, height, /*startRow=*/row, /*numRows=*/1, maxIterations, output);

Q: Is the speedup linear? Explain.

No, the block decomposition is far from linear and is even non-monotonic - 3 threads (1.78×) is slower than 2 threads (1.99×). The Mandelbrot workload is spatially skewed: rows near the vertical center of the image are inside/near the set and cost ~maxIterations per pixel, while rows at the top and bottom escape almost immediately. With 2 threads the top and bottom halves happen to be roughly symmetric, so the split is balanced (~2×). With 3 threads the entire heavy center band lands on the middle thread, which becomes a bottleneck. The interleaved decomposition removes this skew and tracks the ideal line closely (7.54× on 8 threads).

The center threads (3, 4) do ~16× the work of the edge threads (0, 7). Since the phase finishes only when the slowest thread finishes (~96 ms), total speedup is pinned near 4× no matter how many threads are added - exactly what the graph shows. This directly explains the sub-linear block curve.

Q: Describe how you modified the code to improve speedup, and your final 8-thread speedup.

I switched from contiguous bands to an interleaved (cyclic) row assignment: thread t computes rows t, t+P, t+2P, … Because expensive and cheap rows are dealt out round-robin, every thread receives a near-identical mix of work, so per-thread times converge and the phase is no longer bottlenecked by one thread. Final 8-thread speedup: 7.54× (View 1), 7.27× (View 2) - within the target 7-8× band.

Q: What happens with 16 threads? Is there an additional 2× over 8?

No - 16 threads gives only 10.2× (View 1), not ~15×. The machine has 8 physical cores; threads 9-16 run on the second SMT/hyperthread of each core and share the same execution units, L1/L2, and FP pipelines. SMT hides latency but adds little throughput for this compute-bound, ALU-heavy kernel, so the extra 8 threads yield only a modest ~1.35× on top of 8. (On the handout's quad-core, the analogous crossover is at 4→8 threads.)

Program 2 - Vectorizing Code Using SIMD Intrinsics

Approach (clampedExpVector). Process VECTOR_WIDTH elements per iteration. Each lane starts result = 1.0 and multiplies by x exactly y times using a running mask (y > 0); lanes whose exponent is exhausted are masked off while the rest continue. A final masked compare clamps values > 9.999999. The ragged final block is handled with _cs149_init_ones(N - i) so the solution is correct for any N (verified at N = 17, 103, 10000).

Q: Does utilization increase, decrease, or stay the same as width changes? Why?

It decreases as the vector grows wider. The multiply loop runs until the lane with the largest exponent in the block is finished; every other lane sits masked-off for the remaining iterations (this is SIMD divergence). A wider block is statistically more likely to contain a large-exponent element that drags out the whole gang, so the fraction of wasted lanes grows. The instruction count roughly halves each time the width doubles (fewer blocks), but the utilization - useful lane-work ÷ total lane-work - slips downward.

Extra credit - arraySumVector in O(N/VECTOR_WIDTH + log₂(VECTOR_WIDTH)).

Implemented and passing. Phase 1 accumulates all blocks into one wide register with N/W vector adds. Phase 2 tree-reduces the W partial sums to a scalar in log₂W rounds, each round doing one _cs149_hadd_float (sum adjacent pairs) followed by one _cs149_interleave_float (pack the pair-sums into the front half). After log₂W rounds every lane holds the total.

Program 3 - Parallel Fractal Generation Using ISPC

Part 1 - ISPC SIMD (no tasks)

Q: What is the maximum theoretical speedup from 8-wide AVX2? Why is the observed speedup lower?

The gang is 8-wide, so the ceiling from SIMD alone is 8×. Measured: View 1 = 6.05×, View 2 = 5.14×. The gap is SIMD divergence: the 8 pixels in a gang have different iteration counts (some escape after a few iterations, some run to maxIterations), but the gang can only retire when its slowest lane finishes, so lanes that already escaped are wasted. The loss is worst along the fractal boundary and inside the black interior, where neighbouring pixels diverge sharply. View 2 is more divergent overall, giving the lower speedup. Fixed overheads (loop setup, memory) shave off a little more.

Extra credit - std::thread (Prog 1) vs. ISPC tasks: what happens if you launch 10,000 of each?

Program 1 spawns one OS thread per unit. 10,000 std::threads = 10,000 kernel threads on 8 cores → massive oversubscription: context-switch storms, scheduler overhead, and ~10 MB× stacks of memory pressure; performance craters. An ISPC launch[10000] creates 10,000 lightweight tasks placed in a work queue and drained by a small pool of exactly #cores worker threads. Tasks are cheap descriptors, not kernel threads, so 10,000 of them just gives the runtime fine-grained work to load-balance - near-ideal behaviour. Tasks are an abstraction over a thread pool; raw threads are the scheduled resource itself.

Q: Report the SIMD speedup and the multi-core speedup, and quantify each contribution.

On the baseline input, SIMD alone gives ~5.0× and SIMD + tasks gives ~61×. So vectorization contributes ~5×, and multi-core parallelism contributes an additional ~12× on top of that (61 ÷ 5), using 64 tasks spread over 8 cores + SMT. The two axes are roughly multiplicative.

Q: Modify the input to maximize the ISPC (no-task) speedup. Explain.

Set every element to 2.998 - the value that takes the most Newton iterations to converge (note x = 3.0 never converges: the update collapses the guess to 0). Two effects combine: (1) all 8 lanes run the same iteration count → zero divergence, full lane utilization; (2) many iterations per element → the kernel is compute-bound, so per-element and memory overheads are amortized. Result: 6.60×, close to the 8× ceiling (the residual gap is memory traffic for 20M floats plus fixed loop overhead).

Q: Construct an input that minimizes the ISPC (no-task) speedup. Why is 8-wide SIMD inefficient here?

Make one element in every gang of 8 the hard value (2.998) and the other seven trivial (1.0, which converges in zero iterations). The gang still loops to the hard lane's iteration count while the other seven lanes are masked-off the whole time - so the vector unit does the work of a single scalar element per 8 lanes. Result: 0.90×, i.e. slower than scalar, because we pay masking/vector overhead for essentially no parallel work. This is the canonical demonstration that divergence destroys SIMD efficiency.

Q: What speedup do tasks give? Is a large improvement possible? Why or why not?

Only 1.36×. saxpy (result = scale·X + Y) does 2 loads + 1 store and a single FMA per element - an arithmetic intensity of ~0.08 flop/byte. It is memory-bandwidth bound, not compute bound. One or two cores already saturate the DRAM controller, so extra cores/threads add almost nothing. No substantial speedup is achievable on this platform without reducing memory traffic.

Extra credit - why is the bandwidth computed as 4·N·sizeof(float) when only 3 arrays are touched?

Write-allocate. Storing to result[] first triggers a read-for-ownership: the destination cache line must be fetched from memory before it can be (over)written. So the "one store" actually costs one read + one write. Total traffic = load X + load Y + RFO(result) + write(result) = 4·N·sizeof(float). Using non-temporal / streaming stores (e.g. _mm256_stream_ps) skips the RFO, cutting traffic to 3·N and giving up to a 4/3 ≈ 1.33× bandwidth improvement - the realistic "best case" for this kernel.

Program 6 - K-Means Clustering

Measured on a locally-generated dataset (M = 200,000, N = 100, K = 3, ε = 0.1, 51 iterations to convergence), since the official data.dat lives on Stanford AFS. The relative speedups are data-independent.

Q: What is the bottleneck, and how did you optimize it?

Hypothesis: computeAssignments is the hotspot - it is the only O(K·M·N) phase and calls the expensive dist (a pow + sqrt over N dimensions) for every (point, centroid) pair. Profiling confirms it at 68%. Fix: parallelize only this phase across 8 std::threads, partitioned over data points. Because each point is written by exactly one thread, the ranges are disjoint - no locks, no races. I also swapped the loop nesting to point-outer so each point keeps its running minimum in a register (dropping the M-sized minDist scratch array).

Q: Final speedup.

computeAssignments fell from 2529 → 384 ms (6.6× on the phase); total fell from 3705 → 1473 ms = 2.52× overall (target ≥ 2.1×). The clustering output (end.log) is byte-identical to the baseline - the result is unchanged.

Q: There are different axes of parallelism - how did you decide?

Two axes exist: over points (M, ~200k-way) and over centroids (K, only 3-way). I chose points because M ≫ K gives far more parallel slack and perfectly balanced, contention-free partitions; a K-way split would cap at 3× and be imbalanced. I spent the effort on the single phase Amdahl's law rewards most (the 68% one). After the fix, computeCost + computeCentroids now dominate (~77% of the remaining 1.5 s), so parallelizing computeAssignments alone is capped near 1/0.32 ≈ 3.1× overall - consistent with the measured 2.5×.
